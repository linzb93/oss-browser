# 测试用例的编写

针对这个基于 **Vue 3 + Electron + Vite** 的 OSS 客户端项目，使用 `Vitest` 编写测试用例需要采用**分层隔离**和**深度 Mock** 的核心策略。

由于项目同时包含 Node.js 环境（主进程）和浏览器环境（渲染进程），以下是具体的测试编写分析思路：

### 1. 整体测试架构与环境配置

由于 Electron 应用的特殊性，你需要为 Vitest 配置两种不同的测试运行环境（建议通过 Vitest Workspaces 区分）：

- **主进程 (Main Process)**：配置为 `environment: 'node'`。
- **渲染进程 (Renderer Process)**：配置为 `environment: 'jsdom'` 或 `happy-dom`。

### 2. 主进程 (Main Process) 测试方向

主进程负责底层操作和网络通信，是核心业务所在。

- **OSS 业务逻辑抽象**：项目中深度依赖了 `ali-oss`。**绝不要在测试中连接真实的 OSS 桶**。你需要使用 Vitest 的 `vi.mock('ali-oss')` 拦截所有 SDK 调用。重点验证：文件列表分页拉取（处理 marker）、分片上传/下载的切片逻辑、断点续传状态计算，以及网络错误、权限拒绝时的异常抛出与重试机制。
- **本地数据库操作**：项目使用了 `lowdb` 和 `fs-extra` 保存本地配置。在测试中，可以 Mock 文件系统，或将 lowdb 的适配器指向内存（Memory Adapter），以验证账号信息、偏好设置的增删改查逻辑是否正确。
- **IPC 通信监听 (ipcMain)**：主进程通过 IPC 接收前端指令。你需要 Mock `electron` 模块，伪造 `ipcMain.handle` 和 `ipcMain.on` 的触发事件，验证主进程是否正确分发了任务，并正确通过 `event.reply` 返回结果。

### 3. 渲染进程 (Renderer Process) 测试方向

渲染进程主要负责视图呈现和状态维护。

- **组件交互 (@vue/test-utils)**：针对基于 `element-plus` 的文件列表、传输队列、登录配置面板进行组件测试。验证组件能否正确渲染数据源，以及用户的点击、搜索、拖拽等行为是否触发了预期的内部函数。
- **状态管理 (Pinia)**：独立测试 Pinia Store。例如验证“传输列表 Store”中的上传进度计算、任务暂停/取消状态流转、以及多任务并发时的队列维护是否正确。
- **IPC 调用隔离**：前端不应该直接关心后端如何执行。你需要利用 Vitest 全局 Mock 掉 Preload 脚本注入的 `window.electron`（或相关桥接 API），只验证“前端是否在正确的时机，向主进程发送了带有正确参数的请求”。

### 4. 公共纯函数 (Utils & Helpers)

- 项目引入了 `bytes`、`slash` 等工具，自身可能也有路径拼接、大小格式化、日期格式化（`dayjs`）等纯函数。
- 这类函数不依赖特定的系统环境，直接进行传统的单元测试（输入/输出断言），覆盖各种边界情况（如空值、特殊字符、超大数值等）。

### 5. 关键的 Vitest 进阶策略

- **RxJS 异步流测试**：注意到项目中使用了 `rxjs`，涉及复杂的异步任务调度。可以结合 Vitest 的 Fake Timers (`vi.useFakeTimers()`) 来模拟时间推进，测试任务队列的并发控制、轮询逻辑和超时熔断。
- **全局 Setup 文件**：在 `vitest.setup.ts` 中将 `electron` 模块全局 Mock，否则任何引入了 electron 对象的模块在执行测试时都会直接报错崩溃。

这两种方式在前端和 Node.js 社区中都非常常见，它们各有优劣，通常取决于项目的架构风格和团队偏好。

针对你当前这个 **Vue 3 + Vite + Electron** 的项目，以下是具体的分析和建议：

### 1. 同级目录并添加 `.test.ts` / `.spec.ts` 后缀（就近原则）

**形式**：`file.ts` 和 `file.test.ts` 放在同一个目录下。

- **优点**：
    - **高度可见性**：测试文件和源码文件紧密绑定，开发人员在修改源码时，立刻就能看到对应的测试文件，极大地降低了忘记更新测试的概率。
    - **导入极其方便**：测试文件中引入源码只需写 `import { xxx } from './file'`，避免了恶心的 `../../` 多层相对路径。
    - **Vue/Vite 社区绝对主流**：在现代前端开发中，组件、样式、测试文件放在同一个文件夹下（甚至同级）是目前最受推崇的模式。
- **缺点**：如果一个目录下文件极多，源码和测试文件交织在一起，文件列表会显得比较长。

### 2. 在当前目录新建 `__tests__` 目录集中存放

**形式**：在被测试代码所在目录新建 `__tests__` 文件夹，里面存放 `file.test.ts`。

- **优点**：
    - **目录整洁**：源码目录不会被测试文件“污染”，文件列表清爽，业务代码和测试代码有明确的物理隔离。
    - **Jest 遗留习惯**：这是老牌测试框架 Jest 早期非常推崇的模式，很多开发者已经习惯了这种视觉分类。
- **缺点**：
    - 文件导入层级加深，通常需要写 `import { xxx } from '../file'`。
    - 在编辑器目录树中展开/折叠查找对应测试文件时，多了一层目录的点击操作。

### 3. 第三方补充：全局根目录 `tests`

**形式**：在项目根目录（和 `src` 平级）建一个统一的 `tests` 目录，并在里面镜像 `src` 的目录结构。

- **通常不用于单元测试**，而是专门用来存放**端到端测试 (E2E)**（如 Playwright / Cypress）或者**跨多个模块的复杂集成测试**。

---

### 💡 最终建议

对于你的 OSS 客户端项目，**强烈推荐采用第 1 种方式（同级目录 `.test.ts`）为主**：

1.  **Vue 组件和纯函数 (Utils)**：直接在同级建 `*.test.ts`。这能让你在编写 UI 和工具逻辑时拥有最好的心智负担。
2.  **主进程复杂模块 (Main Process)**：如果主进程的某个核心模块（例如处理 OSS 分片上传的逻辑）非常庞大，需要拆分好几个专门的测试文件来测各种边界情况，**局部使用 `__tests__` 目录**将其收纳起来，以保持源码目录的清爽。

**Vitest 的兼容性**：
不用担心配置问题，Vitest 默认的 `include` 规则（`**/*.{test,spec}.?(c|m)[jt]s?(x)`）同时完美兼容这两种存放方式，你甚至可以在项目中混用它们而不需要修改任何配置。
